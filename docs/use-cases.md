# Local image model use cases

Product photography was the stress test for this research, not the only useful
application. The examples below were generated locally on the same RTX 5060 Ti
16 GB. Every input in this page is synthetic and was created for the
demonstration; no private friend, client, or family image is published.

## Private editing and restoration

This use case is personally relevant to the author. His dad has been using AI
to repair damaged historical family photographs for genealogy research, and
the local workflow is being explored as a way to help with that work without
uploading the family archive to a cloud image service. Local execution is also
useful for home interiors, IDs in the background, and other images that should
not casually leave a workstation.

No real family photograph or restoration is included in this repository.
Instead, the demonstration starts with a fictional damaged railway-depot scan
generated by FLUX and restores it with native Qwen Image Edit.

| Synthetic damaged scan | Local Qwen restoration |
|---|---|
| ![A synthetic damaged black-and-white railway depot scan](../assets/sources/restoration-synthetic-damaged-scan.png) | ![The same synthetic railway depot restored into a clean monochrome scan](../assets/results/qwen-restoration-synthetic-scan.png) |

Useful operations include:

- repairing scratches, tears, dust, and fading;
- removing distracting background objects;
- reframing and relighting private interiors;
- producing several crops without repeated cloud uploads.

Restoration should preserve historical evidence rather than silently invent it.
Serious genealogy or archival work should retain the original scan, label the
repaired image as a derivative, compare meaningful details against the source,
and clearly disclose the use of generative repair.

## Synthetic data and evaluation fixtures

Image editing can create controlled lighting, background, camera, weather, or
sensor-condition variants for computer-vision experiments.

| Synthetic reference object | Controlled low-light variant |
|---|---|
| ![A fictional turquoise industrial valve on a neutral studio background](../assets/sources/synthetic-data-valve-reference.png) | ![The same fictional valve under a cool workshop light](../assets/results/flux2-klein-synthetic-data-valve.png) |

Useful applications include:

- bootstrapping an inspection test set;
- testing sensitivity to glare, noise, and low light;
- generating rare-condition fixtures;
- exercising ingestion, thumbnail, and review pipelines.

Synthetic data should supplement real evaluation data. Validate geometry and
class identity, record generation provenance, and keep synthetic samples out of
holdout sets intended to represent reality.

## Confidential design iteration

Early product concepts, packaging, unreleased interfaces, and client campaign
drafts may be useful model inputs but inappropriate cloud uploads.

| Synthetic industrial-design sketch | Local preproduction render |
|---|---|
| ![A marker sketch of a modular headphone charging stand](../assets/sources/confidential-design-headphone-stand-sketch.png) | ![A matte charcoal and cyan render of the charging stand](../assets/results/flux2-klein-confidential-design-headphone-stand.png) |

Local generation supports rapid exploration of color, finish, proportion,
lighting, and presentation while keeping the source inside the chosen network
boundary. The render is still a visual hypothesis, not an engineering drawing.

## Diagrams and presentation visuals

A model can turn a rough composition into a coherent visual layer for a slide,
article, or architecture discussion.

| Rough energy-flow sketch | Clean presentation visual |
|---|---|
| ![A marker sketch connecting a solar house, battery, and electric car](../assets/sources/presentation-energy-flow-sketch.png) | ![A clean isometric energy-flow illustration with space for labels](../assets/results/flux2-klein-presentation-energy-flow.png) |

The image model should not be the authority for labels, dimensions, values, or
citations. Generate the illustration locally, then render text, arrows, legends,
and measurements with deterministic code.

## Offline creative tools

A local model can power a sketch enhancer, storybook tool, photo kiosk, batch
editor, or creative coding environment without network availability, API rate
limits, or per-image metering.

| Locally generated crayon input | FLUX.2 Klein storybook treatment |
|---|---|
| ![A child's crayon drawing of a lighthouse, moon, stars, and sailboat](../assets/sources/offline-creative-lighthouse-drawing.png) | ![A layered paper-and-gouache treatment of the lighthouse drawing](../assets/results/flux2-klein-offline-creative-lighthouse.png) |

The application should call a versioned worker API rather than depend directly
on raw ComfyUI node IDs. The worker can advertise models, free VRAM, maximum
resolution, supported intents, and privacy policy.

## High-volume evaluation and learning

Once the hardware and weights are available, another prompt or seed has almost
no marginal API cost. That makes local generation useful for:

- prompt-ablation and seed-sensitivity studies;
- quantization comparisons;
- workflow regression tests;
- model-upgrade acceptance tests;
- teaching and experimentation.

Local generation does not make brute force automatically wise. It makes broad,
repeatable evaluation economically practical.

## Choosing local, cloud, or hybrid

Use local first when privacy, iteration volume, offline operation, workflow
control, or provenance matters. Use cloud first for low or highly bursty volume,
when a proprietary model is materially better, or when runtime maintenance is
not worth owning. A hybrid system should make routing, cost, and data placement
visible and auditable.
